# 受限 Web3 助手设计

这份设计的目标不是做一个“什么都能自动完成”的链上 Agent，而是明确设计一个**受限 Web3 助手**：它可以帮助用户规划、解释、生成草稿、检查结果和准备交易说明，但不能自动接触私钥、助记词、API Key，也不能绕过人工确认去完成签名、授权、转账或合约写入。

换句话说，这个助手更像一个**高约束的执行前顾问**，而不是一个全权代理。

## 一、这个助手解决什么问题

很多 Web3 初学者会遇到两个问题：

- 看不懂钱包弹窗、合约函数、交易参数和区块浏览器结果
- 想借助 AI 提升效率，但又担心 AI 越权、误导或执行高风险动作

所以这个受限 Web3 助手的目标不是替用户完成链上操作，而是帮助用户：

- 理解即将执行的动作
- 预先检查关键参数
- 生成交互说明、操作清单和记录草稿
- 在执行后帮助核对结果

## 二、用户输入是什么

用户可以给这个助手的输入主要包括：

### 1. 学习型输入

- “帮我解释 `setNumber(uint256)` 这个函数的作用”
- “帮我说明签名和交易的区别”
- “帮我整理测试网交易记录”

### 2. 执行前输入

- 合约地址
- 函数名
- 参数草稿
- 目标网络，例如 `Sepolia`
- 交易目的说明，例如“我要更新合约里的数字”

### 3. 执行后输入

- 交易哈希
- 区块浏览器链接
- 合约地址
- 读取结果
- 钱包弹窗截图中的非敏感信息描述

### 4. 明确禁止的输入

这个助手不应要求，也不应接收以下内容：

- 私钥
- 助记词
- API Key
- token
- `.env` 内容
- 钱包恢复信息
- 任何要求直接代签名或代转账的敏感凭证

## 三、AI / Agent 做什么

这个受限 Web3 助手可以做的事情主要分为五类：

### 1. 概念解释

它可以解释：

- 钱包是什么
- 签名和交易的区别
- Gas 的作用
- 智能合约读写的差别
- 授权为什么有风险

### 2. 交易说明生成

它可以根据用户给出的地址、网络、函数和参数，生成一份**交易前说明**，例如：

- 这次操作的目标是什么
- 预期会发生什么
- 这一步是消息签名、授权还是交易
- 是否会产生 Gas
- 哪些点要特别检查

### 3. 检查清单生成

它可以输出执行前检查清单，例如：

- 网络是否正确
- 目标地址是否正确
- 函数和参数是否正确
- 这一步是否会触发写入
- 是否需要保存交易哈希和区块浏览器链接

### 4. 执行后结果整理

它可以根据用户给出的交易哈希或浏览器链接，帮助用户整理：

- 交易状态
- 区块高度
- Gas 使用量
- 调用目标
- 学习记录草稿

### 5. 风险提示

它可以在每次接近执行边界时提醒：

- 这一步是不是签名
- 这一步是不是授权
- 这一步会不会转账
- 这一步是否会修改链上状态

## 四、Web3 工具或链上步骤是什么

这个助手本身不直接持有钱包控制权，但它可以围绕这些外部工具组织流程：

- 钱包：例如 MetaMask
- 测试网：例如 Sepolia
- 区块浏览器：例如 Sepolia Etherscan
- 合约交互工具：例如 Remix
- 可选脚本工具：例如后续的 viem / wagmi / Hardhat / Foundry

一个典型链上步骤可以是：

1. 用户输入目标任务
2. AI 生成交互说明
3. 用户在 Remix 或前端准备调用
4. 钱包弹窗出现
5. 用户人工确认
6. 测试网执行
7. 用户把交易哈希交回给 AI
8. AI 帮助整理验证记录

## 五、哪些步骤必须人工确认

这是整个设计里最重要的部分。

以下步骤必须人工确认，不能交给 AI 自动通过：

### 1. 地址与参数确认

用户必须亲自确认：

- 目标地址是否正确
- 函数是否正确
- 参数是否符合预期
- 网络是否正确

### 2. 钱包弹窗确认

任何涉及以下动作的钱包弹窗，都必须由人亲自阅读和确认：

- 消息签名
- 代币授权
- 转账
- 合约部署
- 合约写入

### 3. 高风险意图确认

如果操作涉及：

- 金额较大
- 授权范围较大
- 不熟悉的合约
- 主网真实资产

那么必须增加额外人工确认，而不能只凭 AI 的摘要判断。

### 4. 执行结果确认

用户必须亲自去区块浏览器看：

- 交易是否成功
- 调用对象是否正确
- Gas 是否正常
- 是否真的发生了预期写入

## 六、如何验证执行结果

这个助手不把“AI 说成功了”当成成功，而是要求结果验证至少包含以下内容：

### 1. 交易哈希

没有交易哈希，就很难进行公开可验证的链上复盘。

### 2. 区块浏览器链接

这是最直接的验证入口，可以核对：

- 状态
- 区块高度
- 时间
- Gas
- from / to
- 日志与事件

### 3. 合约状态二次读取

如果是合约写入，例如 `setNumber(42)`，还应该再做一次只读检查，例如调用 `getNumber()`，确认链上状态真的被改成预期值。

### 4. 学习记录或 proof-of-work

执行结果最终应沉淀成：

- 交易记录
- 合约地址
- 区块浏览器链接
- 简短说明
- 风险和修正点

这样后续既能提交，也能复盘。

## 七、风险和限制

### 1. AI 可能解释错

即使助手能生成解释，它也可能误读函数、参数、授权范围或执行结果。因此它只能作为辅助说明层，不能替代真实确认。

### 2. AI 不能看见完整钱包安全上下文

AI 不应该接触私钥、助记词和敏感凭证，因此它也不可能真正替用户承担钱包级安全责任。

### 3. 自动化边界必须保持收敛

这个助手被设计成“受限”，就是为了避免它从说明工具滑向高风险执行工具。它可以准备交易说明，但不能自动签名；可以整理结果，但不能绕过确认去写链上状态。

### 4. 浏览器和钱包提示仍然是权威入口

真正的签名内容、授权内容和交易内容，以钱包弹窗和链上回执为准，不以 AI 文本总结为准。

### 5. 不适合高风险主网自动化

这个设计更适合：

- 学习环境
- 测试网
- 小额演示
- 文档辅助

它不适合在没有严格风控和工程审计的情况下，直接扩展成真实主网资金自动化系统。

## 八、最小工作流示例

可以把这个受限 Web3 助手理解成下面这条流程：

`用户提出目标 -> AI 解释和生成说明 -> 用户检查地址/函数/参数 -> 钱包人工确认 -> 测试网执行 -> 区块浏览器验证 -> AI 整理学习记录`

在这条流程里：

- AI 负责理解、解释、整理和提醒
- 人负责授权、签名、判断和承担最终责任
- 区块链负责执行
- 区块浏览器负责公开验证

## 九、总结

这个受限 Web3 助手最重要的不是“它能做多少事”，而是“它明确不能做什么”。

它适合做：

- 解释
- 草稿生成
- 参数检查清单
- 结果整理
- 风险提醒

它不能做：

- 接触私钥或助记词
- 保存敏感凭证
- 绕过人工确认签名
- 自动完成授权、转账或合约写入

如果用一句话总结它的设计原则，就是：

**AI 负责降低理解成本，人保留执行控制权，链上结果必须公开可验证。**
