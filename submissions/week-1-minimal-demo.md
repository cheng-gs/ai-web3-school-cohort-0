## 1. 它解决什么学习问题

这个 demo 解决的是“学员知道术语，但说不清楚，也不知道怎么把概念转成行动”的问题。

Week 1 同时覆盖 AI 和 Web3 基础，容易出现这些情况：

- 听过 `agent`、`workflow`、`wallet`、`signature`、`gas` 这些词，但理解停留在表面
- 能复述定义，但不会结合自己的背景理解
- 不知道一个概念的常见误区和风险边界

所以我做了一个最小概念解释器：让用户输入概念、背景和困惑，再输出一份结构化学习草稿。

## 2. 用户如何与它交互

用户交互流程很简单：

1. 打开本地网页 `index.html`
2. 选择一个概念，例如 `Signature`
3. 选择自己的背景，例如“我偏 AI，Web3 刚入门”
4. 选择输出风格
5. 输入当前的困惑
6. 点击“生成解释草稿”

页面会输出：

- 一段解释
- 一个具体例子
- 一个类比
- 一个误区或边界提醒
- 一个针对当前困惑的说明
- 三条下一步行动建议

## 3. 输入示例和输出示例

### 输入示例

- 概念：`Signature`
- 背景：`我偏 AI，Web3 刚入门`
- 输出风格：`多用对比和类比`
- 困惑：`我总把签名和交易混在一起，也分不清钱包弹窗到底是在登录、授权，还是在真的发交易。`

### 输出示例

输出会生成一份学习草稿，核心内容包括：

- 解释：签名是用私钥对消息或动作做授权证明，不等于已经把交易发到链上
- 例子：很多 dApp 先要求消息签名验证身份，这和真正发送交易不是同一步
- 类比：签名像在文件上盖章同意，但盖章不等于文件已经进入正式执行系统
- 误区：不能把所有钱包弹窗都理解成“只是登录”
- 下一步：下次看到钱包弹窗时，先区分它是在请求消息签名、授权，还是实际交易

## 4. 哪部分由 AI 生成，哪部分由我人工修改 / 验证

### AI 辅助的部分

- demo 的整体形式选择
- 第一版页面结构
- 第一版概念解释模板
- 第一版示例文案草稿

### 我人工修改 / 验证的部分

- 把文档和页面统一改成中文
- 调整概念解释，让它更适合“AI 熟悉、Web3 新手”的学习背景
- 修正边界表达，例如：
  - 钱包不是普通 Web2 登录
  - 签名不等于交易
  - 工具调用不等于可靠自动化
  - 涉及签名、授权、转账和写入时必须保留人工确认
- 检查页面输出是否真的能帮助后续写 daily note 或打卡草稿

## 5. 限制和下一步改进

当前限制：

- 这是一个本地静态网页，没有接真实模型 API
- 输出是“AI 辅助生成结构 + 模板化生成结果”，不是真正的实时模型回答
- 当前概念数量有限，主要覆盖 Week 1 的入门场景
- 还没有保存用户历史记录，也没有接入 GitHub 或 daily note 自动写入

下一步改进方向：

1. 增加更多概念，例如 `Contract`、`Testnet`、`Block Explorer`、`Guardrails`
2. 增加一个“AI × Web3 流程模式”，直接生成 `AI 输出 -> 人工复核 -> 钱包确认 -> 链上执行 -> 浏览器验证` 的解释卡片
3. 接入真实模型 API，让它根据用户问题生成更个性化的解释
4. 增加“导出到 Markdown”功能，方便直接写入 daily note 或提交记录
5. 给每个概念增加官方资料链接和验证入口，降低误导风险
