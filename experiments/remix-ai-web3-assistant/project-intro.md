# Remix + AI Web3 助手网页介绍

## 这个网页是做什么的

这个网页是我当前 AI × Web3 项目的第一阶段 MVP，也就是一个**面向 Remix 测试网练习的受限 AI 助手页面**。

它的目标不是替用户自动完成链上操作，而是帮助用户在真正去 Remix 和 MetaMask 执行之前，先把这次操作理解清楚、准备清楚、记录清楚。

换句话说，这个网页负责的是：

- 解释
- 提醒
- 整理
- 生成草稿

而不是：

- 自动签名
- 自动部署
- 自动转账
- 自动写链上状态

## 为什么要做这个网页

在 AI × Web3 的学习和实践里，一个很常见的问题是：

- 用户知道大概要去 Remix 点哪里
- 但不确定这一步到底是读取、写入、授权还是部署
- 也不确定钱包弹窗为什么出现
- 更不知道执行后应该记录哪些材料，才能形成可验证的 proof-of-work

所以这个网页的意义是把一条原本分散的流程先理顺：

`用户输入目标 -> AI 生成说明 -> 用户去 Remix 操作 -> 钱包人工确认 -> 测试网执行 -> 用户回填结果 -> AI 整理记录`

它解决的不是“自动执行”的问题，而是“在执行前后减少理解成本和记录成本”的问题。

## 这个网页现在能做什么

当前这个网页主要支持两类能力。

### 1. 链上交互说明

用户输入：

- 目标网络
- 合约名称
- 合约地址
- 函数名
- 参数
- 操作类型
- 钱包类型
- 用户目标描述

网页会生成：

- 这次操作的解释
- 这是只读、写入还是部署
- 是否需要钱包确认
- 是否会消耗 Gas
- Remix 操作步骤
- 风险提醒
- 执行前检查清单
- 执行后 Markdown 记录模板

这部分的作用是：帮助用户在真正点击 Remix 和 MetaMask 之前，先知道自己要做什么、为什么做、哪里要小心。

### 2. 自然语言修改合约

用户输入：

- 合约名称
- 自然语言修改需求
- 原始 Solidity 合约代码

网页会生成：

- 修改摘要
- 改动清单
- 风险提醒
- 人工复核清单
- Remix 下一步
- 修改后的 Solidity 草稿

这部分的作用是：让 AI 先根据自然语言需求给出一版修改草稿，再由人把草稿拿去 Remix 编译、检查和决定是否部署。

## 这个网页和 DeepSeek API 的关系

这个网页现在已经支持通过本地后端接入 DeepSeek API。

整体结构是：

`浏览器页面 -> 本地后端 -> DeepSeek API`

这样设计的原因是：

- API Key 不会暴露在前端页面里
- API Key 只保存在本地 `.env.local`
- 前端页面不直接接触密钥

所以这个网页已经不是纯静态模板，而是一个：

- 前端负责收集输入和展示结果
- 本地后端负责读取环境变量并调用模型 API

的最小可运行系统。

## 这个网页的安全边界

这个网页是一个**受限 Web3 助手**，所以它有明确边界。

### 它可以做的事

- 解释函数和链上动作
- 判断只读 / 写入 / 部署的区别
- 生成操作说明
- 生成执行前检查清单
- 生成执行后记录模板
- 根据自然语言需求改写 Solidity 草稿

### 它不能做的事

- 不能接触私钥
- 不能接触助记词
- 不能接触钱包恢复信息
- 不能替用户签名
- 不能替用户点击 MetaMask 确认
- 不能绕过人工确认自动授权
- 不能自动转账
- 不能自动部署到链上

这条边界非常重要，因为它决定了这个项目现在的定位是：

**AI 辅助理解和准备链上操作，而不是自动控制链上执行。**

## 这个网页适合什么场景

它最适合这些场景：

- 学习测试网交易和合约交互
- 使用 Remix 做最小合约练习
- 在执行前确认风险和步骤
- 在执行后整理 proof-of-work
- 用自然语言先生成一版合约修改草稿

特别适合像我现在这样的阶段：

- 已经开始做测试网和 Remix 实践
- 想把 AI 和 Web3 结合起来
- 但又不希望一开始就做一个过度自动化、风险过高的 Agent

## 这个网页目前的限制

虽然它已经能运行，但它仍然是第一阶段 MVP，所以限制也很明确：

- 不直接连接 MetaMask
- 不直接读取链上状态
- 不自动获取交易哈希
- 不自动部署合约
- 不自动执行写入交易
- 不自动保存链上结果到仓库

另外，合约修改草稿虽然能由 AI 生成，但仍然必须经过：

- 人工阅读
- Remix 编译
- 人工检查访问控制和副作用
- 人工决定是否测试或部署

## 我为什么先做这个版本

因为这个版本是打通全链路的最快方式。

如果一开始就做：

- 前端页面
- 钱包连接
- `wagmi`
- `viem`
- 页面内直接读写
- 状态管理
- 错误处理

那工程复杂度会明显上升，可能在真正验证需求之前就被技术细节拖住。

而现在这个第一阶段版本先验证的是：

- AI 的解释有没有帮助
- 风险提醒有没有价值
- 记录模板是否真的节省整理时间
- 用户是否更容易理解链上操作边界

只有这些答案是肯定的，第二阶段的产品化才值得做。

## 下一步如何升级

这个网页的下一步升级方向已经比较明确：

### 第二阶段目标

把“解释、执行、验证记录”尽量收进同一个网页里。

### 可能的技术方向

- Next.js
- wagmi
- viem
- 页面内连接 MetaMask
- 页面内直接读函数
- 页面内发起写入交易
- 自动拿到交易哈希
- 自动生成浏览器链接和记录

### 但仍然保留的边界

即使升级到第二阶段，也仍然要坚持：

- AI 不接触私钥
- 钱包确认必须人工完成
- 授权、转账、部署、写入不能绕过人工确认

## 总结

如果用一句话总结这个网页的作用，那就是：

**它是一个面向 Remix 测试网练习的受限 AI 助手，负责帮助用户理解、准备和记录链上操作，但不替用户执行高风险链上动作。**

这也是我当前把 AI 和 Web3 结合起来的最小可行路径。
